# AI Business Assistant

RAG-based agent: company documents → local FAISS → OpenAI answers (planned). This repo is being built in phases.

## Phase 1 — Project skeleton (done)

**Delivered**

- FastAPI app with `GET /health` only (`backend/app/main.py`).
- Package layout: `backend/app/api/`, `services/`, `rag/` (empty except `__init__.py`).
- Streamlit app shell (`frontend/app.py`; chat UI completed in Phase 6).
- Dependencies: `requirements.txt`.
- Secrets: `.env.example` template; `.env` for local keys (gitignored). Phase 2 loads `.env` from the project root.

**Not in scope for Phase 1**

- RAG, uploads, embeddings, FAISS, OpenAI calls, or Streamlit ↔ API wiring.

## Phase 2 — LLM integration (stateless chat) (done)

**Delivered**

- `backend/app/config.py` — loads project-root `.env` on import (used by `main` before routes run).
- `backend/app/services/llm.py` — `AsyncOpenAI` chat completion (single user turn; shared system prompt). Optional **Gemini fallback** when OpenAI fails with quota / rate-limit style errors (e.g. HTTP 429, `insufficient_quota`), if `GEMINI_API_KEY` is set. No conversation memory.
- `POST /chat` — JSON `{ "message": "..." }` → `{ "reply": "..." }` (Phase 2: plain LLM; **Phase 5 onward: RAG**, see below).
- Env: `OPENAI_API_KEY` (required for the primary path). Optional `OPENAI_CHAT_MODEL` (default `gpt-4o-mini`). Optional `GEMINI_API_KEY` and `GEMINI_CHAT_MODEL` (default `gemini-2.0-flash`) for fallback.

**Not in scope for Phase 2**

- Vector store, RAG, document upload, chat history, Streamlit calling the API.

### Chat API example

```powershell
curl -X POST "http://127.0.0.1:8000/chat" ^
  -H "Content-Type: application/json" ^
  -d "{\"message\": \"What is EBITDA in one sentence?\"}"
```

OpenAPI docs: `http://127.0.0.1:8000/docs` — this UI is **Swagger UI**, bundled with FastAPI from your OpenAPI schema. It is not Postman; it is generated by FastAPI so you can try endpoints in the browser (similar idea to Postman, but built-in and maintained by the framework).

### OpenAI quota errors + Gemini fallback

If OpenAI returns **429** / **insufficient_quota**, set `GEMINI_API_KEY` in `.env`. The same `POST /chat` request will then answer via Gemini when the OpenAI call hits that class of error.

**Security:** never commit API keys. If a key was pasted into chat, email, or a ticket, **rotate** it in the provider console and update `.env` only on your machine.

## Phase 3 — Document ingestion (done)

**Delivered**

- `POST /upload-document` — multipart field `file`: **.pdf**, **.txt**, or **images** (PNG, JPG, JPEG, WebP, GIF; max ~15 MB). Images are vision-captioned then chunked like text.
- Text extraction: UTF-8 **TXT**; **PDF** via `pypdf` (per-page text, joined).
- Chunking: fixed windows with overlap (defaults **1000** chars, **150** overlap); tunable with `DOCUMENT_CHUNK_MAX_CHARS` and `DOCUMENT_CHUNK_OVERLAP` in `.env`.
- In-memory storage: `document_id` → list of chunk strings in the API process (lost on restart; no vector DB).

**Response:** `{ "chunk_count": <int>, "document_id": "<uuid>" }`.

**Not in scope for Phase 3**

- Embeddings, FAISS, retrieval, or wiring chunks into `/chat` (RAG wiring is Phase 5).

### Upload example

```powershell
curl -X POST "http://127.0.0.1:8000/upload-document" ^
  -H "accept: application/json" ^
  -F "file=@C:\path\to\document.pdf"
```

Or use **Swagger UI** at `/docs` → **POST /upload-document** → **Try it out** → choose file.

## Phase 4 — Vector store (FAISS) (done)

**Delivered**

- `backend/app/services/embedding_service.py` — OpenAI **text embeddings** (`text-embedding-3-small` by default); batched requests; vectors **L2-normalized** for cosine-style search via inner product.
- `backend/app/services/vector_store.py` — **FAISS** `IndexFlatIP`, row-aligned metadata (`document_id`, `text`), **persist** under `data/vector_store/` (`faiss.index`, `meta.jsonl`). Loads existing index on app **startup** (`lifespan`).
- `retrieve_relevant_chunks(query, top_k)` — async search returning scored chunk hits.
- **API**
  - `POST /vector/index` — JSON `{ "document_id": "<uuid from upload>" }`: embeds in-memory chunks for that document, appends to the index, saves to disk. Returns `{ "indexed_chunks", "total_vectors" }`.
  - `POST /vector/search` — JSON `{ "query": "...", "top_k": 5 }`: returns `{ "results": [ { "document_id", "text", "score" } ] }` (higher **score** = more similar).

**Env:** optional `OPENAI_EMBED_MODEL` (default `text-embedding-3-small`). Do not change embedding model after building an index unless you delete `data/vector_store/` and re-index (dimension / space must stay consistent).

**Git:** `data/` is gitignored (local index only).

**Not in scope for Phase 4**

- Wiring retrieval into `/chat` (done in Phase 5), Streamlit (Phase 6), Gemini embeddings.

### Vector flow example

1. `POST /upload-document` → note `document_id` and `chunk_count`.
2. `POST /vector/index` with that `document_id`.
3. `POST /vector/search` with a natural-language `query`.

## Phase 5 — RAG pipeline (done)

**Delivered**

- `backend/app/services/rag.py` — `answer_with_rag(question)`: retrieves top-`RAG_TOP_K` chunks from FAISS (same path as `/vector/search`), dedupes exact duplicate excerpts, calls the LLM with injected context.
- `backend/app/services/llm.py` — `complete_rag_turn(context_chunks, question)`: strict **system** prompt requiring answers to be supported only by the supplied excerpts; **user** message contains numbered excerpts + question (or an explicit “no retrieval” instruction when the index is empty or nothing matches).
- **`POST /chat`** — now runs the **full RAG flow** (retrieval → grounded generation). Optional env **`RAG_TOP_K`** (default **5**, max **20**).

**Prerequisites for useful answers**

1. `POST /upload-document` → `document_id`
2. `POST /vector/index` with that id (builds embeddings + FAISS)
3. `POST /chat` with `{"message": "..."}`

If nothing is indexed or retrieval finds no neighbors, the model is instructed **not** to invent document content and to say that no relevant excerpts were found.

**Not in scope for Phase 5**

- Streamlit UI (Phase 6), citation spans, multi-turn memory.

## Phase 8 — Vision & media (chat + PDF figures)

**Delivered**

- **`POST /chat`** accepts JSON `message` (optional if images present) plus `images[]` with `{ "media_type", "base64_data" }` (JPEG/PNG/WebP/GIF, size-capped). RAG still retrieves text chunks; the LLM receives **document excerpts + optional user images** (OpenAI vision; Gemini multimodal on quota fallback when images are present).
- **`backend/app/services/vision_service.py`** — for **PDF uploads**, embedded raster images are extracted (**PyMuPDF**), briefly **captioned** with the vision model, and the captions are appended to the extracted text before chunking/indexing so charts/figures contribute to search.
- **Streamlit** — file picker for **photos or short video**; **video → first frame** (PNG) via **imageio**; **Send attachment only** when no typed message; widget resets after each send.
- **Dependencies:** `pymupdf`, `Pillow`, `imageio`, `imageio-ffmpeg` (for common MP4/WebM first-frame extraction).

**Limits / caveats**

- Video is **not** analyzed as full motion—only an approximate **first frame** is sent as an image. Large or exotic codecs may fail; prefer short clips or screenshots.
- PDF figure handling depends on embedded images; vector graphics without raster images may not produce captions.
